## 2. 이번 버전 개발 완료

### 일정 화면 모듈 분리

- `frontend/src/screens/ScheduleScreens.jsx`를 추가했다.
- 반복 거래 관리와 할부 일정 화면을 일정 도메인 모듈로 묶었다.
- App은 규칙·할부 데이터와 생성·수정·삭제 callback을 전달하고 화면 모듈이 표시와 입력 이벤트를 담당한다.

### 반복 거래 화면 분리

- 반복 거래 생성·수정 폼을 `ScheduleScreens.jsx`로 이동했다.
- 수입·지출·이체 유형 전환과 자산·카테고리 선택을 함께 이동했다.
- 금액, 내용, 반복 주기, 간격, 시작일, 다음 실행일, 종료일 입력을 유지했다.
- 오늘분 생성, 새 규칙 초기화, 기존 규칙 수정·삭제 UI를 이동했다.
- 반복 주기 표시 helper와 거래 유형별 색상 표시를 모듈 내부에서 관리한다.

### 반복 거래 초기 폼 분리

- `emptyRecurringForm`을 일정 모듈에서 export하도록 변경했다.
- 기본 날짜를 함수 인자로 받을 수 있고, 생략 시 현재 날짜를 사용한다.
- App의 상태 초기화와 화면의 새 규칙 초기화가 같은 factory를 사용한다.

### 할부 일정 화면 분리

- 할부 그룹 총액·개월 요약과 회차별 일정 목록을 일정 모듈로 이동했다.
- 현재 선택 회차 강조, 전체 수정 진입과 전체 삭제 callback을 유지했다.
- 공통 `Metric`, `AppHeader`, `BackButton`, `EmptyState`를 재사용했다.

### App import 정리

- 일정 화면 이동 후 `main.jsx`에서 사용하지 않는 공통 UI와 포맷 helper import를 제거했다.
- `main.jsx`는 일정 데이터 로드·저장과 패널 전환 책임만 유지한다.

### 파일 규모 변화

- `main.jsx`는 약 1,334줄에서 약 1,151줄로 감소했다.
- 반복 거래·할부 화면과 초기화 helper 약 200줄을 `ScheduleScreens.jsx`로 이동했다.
- 프론트엔드 빌드 모듈 수는 39개로 증가했다.

## 4. 아직 부분 완료인 항목

### 프론트엔드 구조

- 예산 설정 화면과 하단 탐색·앱 셸은 아직 `main.jsx`에 남아 있다.
- API 요청과 상태 로직을 custom hook 또는 API 모듈로 분리하는 작업은 아직 미완료다.
- App 컴포넌트가 여러 도메인의 API 호출과 패널 상태를 함께 관리한다.

### 백엔드 구조

- `LedgerService`가 거래, 검색, 예산, 통계, 명의와 자산 관리 책임을 함께 담당한다.
- 도메인별 서비스 분리와 검색 조건 객체 분리가 필요하다.

### 기타 후속 항목

- 검색 결과 페이지네이션과 정렬 선택은 아직 미완료다.
- 영수증 첨부 파일 정리 정책 고도화는 아직 미완료다.

## 7. 개발 메모

- 반복 거래와 할부는 모두 미래 일정과 회차를 다루는 화면이므로 하나의 일정 모듈에 배치했다.
- 저장·삭제 정책과 API 호출은 App에 유지하고 화면은 props 기반으로 분리해 기존 상태 흐름을 보존했다.
- 초기 폼 factory를 화면과 App이 공유하도록 해 새 규칙과 저장 후 초기화 값이 달라지는 문제를 방지했다.
